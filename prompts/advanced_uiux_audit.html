1 · Basic Functionality Category 15 issues

The application handles the happy path well but has zero resilience infrastructure for real-world conditions. There is no offline indicator, no idle state, no concurrent-action guard, and no intelligent clipboard handling. SWR's revalidateOnFocus is enabled on ~8 of 36 data pages — the remaining 28 serve stale data silently after background tab use.
Sub-CriterionScreen / FileIssueSeverityFix
Offline Indicator Global
layout.js		 No offline/online state detection anywhere. When a user on a college campus with spotty WiFi loses connection, SWR silently fails, skeletons persist, and the user has no indication the problem is network-related vs a server error. High Add a useOnlineStatus hook using navigator.onLine and window.addEventListener('online'/'offline'). Show a fixed bottom banner: "You're offline — changes won't save until reconnected." On reconnect, call mutate() on all active SWR keys and dismiss the banner.
Background Tab Staleness 28 SWR pages
(non-employer pages)		 revalidateOnFocus: true is set on only employer-role pages (applications, drives, jobs, internships). The remaining 28 SWR pages (college dashboard, student dashboard, reports, clarifications, etc.) use default SWR settings — they do not refresh when the user returns to the tab after 5+ minutes. High Set revalidateOnFocus: true globally via SWR's <SWRConfig value={{ revalidateOnFocus: true, revalidateOnReconnect: true }}> in the root layout. Remove per-page overrides. Apply refreshInterval: 120000 (2 min) on high-frequency pages: applications, notifications, drives.
Idle Session State Global
All authenticated pages		 After 5+ minutes of inactivity, the UI shows no visual change. The 24hr session continues silently. There is no auto-save notice, no dim overlay, and no proactive warning that the session will eventually expire. A TPO who leaves a partially-filled drive form for 30 minutes returns to a session that is now closer to expiry with no indication. Medium Add an idle detector using mousemove / keydown / touchstart events with a 20-minute threshold. After idle, show a subtle toast: "Still there? Your session is active." This doubles as a session health signal. For forms with unsaved data, show a more prominent "You have unsaved changes" banner.
Concurrent Action Blocking All mutation buttons
Forms across app		 Submit buttons are disabled while submitting state is true, but there is no protection against race conditions from rapid sequential clicks before the state updates. On mobile with tap-bounce, a user can fire two create-job or create-drive requests in under 100ms — creating duplicate records in the database. Medium Add a useRef-based in-flight guard that cannot be bypassed by React re-render timing: const pending = useRef(false); if (pending.current) return; pending.current = true; try { await submitFn(); } finally { pending.current = false; }. This prevents the race condition even before the submitting state prop propagates.
Clipboard Handling Login · Register · Forms
Email/phone inputs		 Email fields do not auto-trim pasted whitespace. Pasting " sandeep@college.edu " (with leading/trailing spaces) passes silently through the UI, hits the API, fails with a non-obvious error, and the user cannot see the spaces in the input field. This is the most common cause of "login not working" support tickets. Medium Add an onPaste handler to all email inputs: onPaste={(e) => { e.preventDefault(); const text = e.clipboardData.getData('text').trim(); setEmail(text); }}. Alternatively, add .trim() in the onChange handler. Apply to registration, login, and all admin user-creation forms.

2 · Interaction Category 25 issues

The button press scale (transform: scale(0.97)) and card hover lift (translateY(-2px)) are genuinely polished micro-interactions. The failure is in feedback latency tiers — every action (toggle, status update, bulk approve) waits for full network round-trip before any visual change. Interaction memory is almost entirely absent: no view preference, no form state, and no scroll position is remembered.
Sub-CriterionScreen / FileIssueSeverityFix
Feedback Latency Tiers All mutation flows
Status toggles, form submits		 Every action uses a single feedback tier: wait for server confirmation, then update. No action uses instant visual feedback (toggle flips immediately) or predicted state (skeleton replaces content while waiting). Users experience 200–800ms of ambiguity on every click. At scale on a shared campus network, this will routinely exceed 1 second. High Introduce three tiers: (1) Instant: pure UI state — toggle switches, tab changes, expand/collapse flip immediately. (2) Predicted: SWR optimistic update with rollback on failure — job publish, application status change. (3) Confirmed: form creates that require ID from server. Tier assignment based on reversibility of the action.
Interaction Memory Tables with view toggle
college/drives, employer/jobs		 College drives and employer interviews have a list/calendar view toggle (setView('list' | 'calendar')). This preference resets to 'list' on every page load. A user who prefers the calendar view must switch it every single visit. No interaction preference (view mode, expanded sections, panel width) persists across sessions. High Persist view preference to localStorage with a page-scoped key: localStorage.setItem('drives.view', view). On mount: const [view, setView] = useState(() => localStorage.getItem('drives.view') || 'list'). Apply to: drives (list/calendar), interviews (list/calendar), reports (chart/table). Extend to collapsed sidebar sections.
Disabled State Clarity All disabled buttons
Forms, action menus		 Most disabled buttons have no explanation. The "Save Draft" button is disabled while the form is empty, but hovering gives no tooltip. The "Publish" button in college infrastructure is not shown (hidden entirely) when the user lacks permissions — instead of being visible but disabled with an explanation. High For disabled buttons: always show, never hide. Add title="Reason why disabled" as a minimum — e.g., title="Add at least one campus to enable publishing". For permission-blocked actions: show disabled with title="You need admin permissions to do this". One instance of this done correctly already exists on college/employers "No website on file" — replicate the pattern.
Escape Key Handling Notification panel, forms
layout.js, inline forms		 The notification dropdown and all inline forms (add interview, add job, add event) have no Escape key handler. Once open, the only way to close them is clicking a specific button or target area. On keyboard navigation, the user is trapped without Escape. High Add a useEffect that listens for keydown with key === 'Escape' whenever a panel or form is open: close the panel and return focus to the trigger button. This is a single reusable hook: useEscapeKey(isOpen, onClose). Apply to: notification panel, mobile sidebar, all inline create forms.
Hover Intent Delay Action button menus
Sidebar items with sub-items		 Dropdown menus and action panels open immediately on mouseEnter with no delay. On desktop, users moving the cursor across the page will accidentally trigger dropdowns. There is no 100–150ms hover intent delay to distinguish intentional hover from cursor travel. Low Add a useHoverIntent hook that uses setTimeout to delay showing the dropdown by 120ms after mouseEnter. Cancel the timer on mouseLeave. This prevents accidental triggers while making intentional hover feel instant. Apply to all dropdown triggers and sidebar nested groups.

3 · Navigation Category 34 issues

The sidebar correctly highlights the active route via pathname === item.href. Beyond that, navigation UX is thin: filter state is lost on every back-navigation, scroll position is never preserved, modals have no Escape key or close affordance, and anchor links have no arrival feedback.
Sub-CriterionScreen / FileIssueSeverityFix
Context Persistence All filtered tables
students, applications, jobs		 Navigating away from a filtered table (e.g., applications filtered to "Shortlisted + CSE") and pressing browser Back resets all filters to default. The user must reapply filters every time. There is no visual indicator that the list was previously filtered. High Serialise filter state into URL query params: ?status=shortlisted&dept=CSE. Next.js Router's useSearchParams / useRouter.replace can update URL without page reload. On mount, hydrate filter state from URL. Browser Back then restores the correct filter set automatically.
Scroll Position Memory All list pages
drives, students, jobs		 Scrolling to position 800px in the students table, clicking a student action, then pressing Back returns the user to the top of the list. With 200+ students, the user must re-scroll to find their position. This is a high-friction pattern for TPOs who process students one-by-one. Medium Use Next.js's scroll: false option on programmatic navigation: router.push(href, { scroll: false }). For full-page navigations that reload data, store scroll position in sessionStorage before navigating and restore it after data loads: window.scrollTo(0, savedPosition).
Modal Navigation Trap Inline forms (create job, add event)
Notification panel		 Inline forms and the notification dropdown open without trapping focus or clearly indicating that background content is inaccessible. Users navigating by Tab can exit the open panel and interact with background page elements while the panel is still open — creating a disorienting split context. Medium For panels that overlay content, add a transparent click-outside dismissal backdrop. For modal-style forms, implement focus trapping: tab cycles only within the open form. Add a visible × close button. These are low-effort: onKeyDown + tabIndex management in a useFocusTrap hook.
Anchor Link Feedback Reports page tabs
Multi-section forms		 The college reports page uses tab navigation to switch between sections. When switching tabs, there is no scroll-to-section behaviour, no highlight animation on the newly active content, and no scroll-padding-top set to account for the sticky topbar — so anchors scroll behind the topbar on direct link. Low Add scroll-padding-top: var(--topbar-height) to the html element in globals.css. On tab switch, briefly apply a highlight class to the incoming section: animate-highlight { animation: highlight 0.8s ease-out; } where the animation fades from a light background color to transparent.

4 · Accessibility — Beyond WCAG Basics Category 45 issues

2 of 5 advanced accessibility criteria are now implemented — category grade lifted from F to C. ✓ Reduced-motion support is live in globals.css (@media (prefers-reduced-motion: reduce) covers all animations). ✓ Focus ring visibility is implemented for buttons, inputs, selects, table rows, and export controls via :focus-visible rules. Remaining open gaps: dynamic content focus management, screen reader announcement granularity (aria-live), and alternative bypass landmarks.
Sub-CriterionScreen / FileIssueSeverityFix
Reduced Motion ✓ FIXED globals.css
All animated elements		 Resolved. @media (prefers-reduced-motion: reduce) block added to globals.css. Covers all hover transforms, skeleton shimmer, entrance animations (animate-fadeIn, animate-slideUp), and card lift effects. Users with vestibular disorders no longer receive unwanted motion. WCAG 2.3.3 compliance achieved. Fixed Implemented — single @media (prefers-reduced-motion: reduce) block in globals.css sets animation-duration: 0.01ms and transition-duration: 0.01ms on all elements. Skeleton shimmer gracefully degrades to static background.
Dynamic Content Focus Inline forms (add job, add event)
Search results, load more		 When inline create forms open (the job creation form on employer/jobs, the event form on college/calendar), focus is not moved to the first field. Keyboard users must Tab through the entire page to reach the newly injected form content. When the form closes, focus is not returned to the trigger button. High On form open: useEffect(() => { if (showForm) firstFieldRef.current?.focus(); }, [showForm]). On form close: triggerButtonRef.current?.focus(). For dynamically loaded content (load more, search results), move focus to the first new item: firstNewItemRef.current?.focus() after the DOM update settles.
Announcement Granularity ToastProvider, error states
Global feedback system		 All screen reader announcements use the same priority level — or none at all. Success toasts, error alerts, and status updates are all regular div elements with no aria-live attribute. Screen readers are silent on all feedback. For a multi-role system where errors (application rejected, offer deadline missed) are time-sensitive, silent feedback is a critical accessibility failure. High Use two live region priorities: role="alert" aria-live="assertive" for errors, timeouts, and destructive confirmations (interrupts screen reader immediately). role="status" aria-live="polite" for success messages and info updates (waits for current speech to finish). Update ToastProvider to set the correct role based on type: 'error' | 'success' | 'info'.
Focus Ring Visibility ✓ FIXED globals.css
.btn, .form-input, all interactive		 Resolved. :focus-visible rules added to globals.css for: .btn, .form-input, .form-select, .form-textarea, .student-row-clickable, .export-csv-primary, .export-csv-trigger, .export-csv-chevron-btn, and .export-csv-menu-item. Keyboard users now get a visible 2px solid outline on all interactive elements. Fixed Implemented — .btn:focus-visible { outline: 2px solid var(--border-focus); outline-offset: 2px; } and equivalent rules for inputs, selects, table rows, and export controls. WCAG 2.4.7 Non-text Contrast requirement met.
Alternative Bypasses Root layout
All pages		 There is no skip-to-main-content link (covered in SaaS audit), but also no way to skip repetitive table pagination controls, long filter bars, or the calendar grid's 31 day cells. A screen reader user navigating the college calendar must Tab through every cell to reach the "Add Event" button at the bottom. Medium Add landmark roles: role="main" on the content area, role="navigation" on the sidebar. Add region landmarks for tables: role="region" aria-label="Students table". Screen reader users can then jump between landmarks. For the calendar, add a visible "Jump to actions" link before the grid that skips to the action buttons.

5 · Analytics Category 53 issues

Zero analytics tracking exists in the application. No page views, no event tracking, no session replay, no frustration signal capture. For a SaaS platform with 4 roles and 55 pages, the product team has no visibility into which features are used, where users drop off, or what causes frustration. This is a complete blind spot.
Sub-CriterionScreen / FileIssueSeverityFix
No Event Tracking Global
All pages		 No analytics library is integrated (no Google Analytics, Plausible, PostHog, Mixpanel, or equivalent). There is no tracking of page views, button clicks, form submissions, or feature adoption. The product team cannot answer: "How many employers post more than one job?" or "Which page has the highest exit rate?" High Integrate a privacy-respecting analytics tool — Plausible or PostHog (both GDPR-compliant, self-hostable). Add page-view tracking via Next.js Router events. Add key event tracking for: job posted, drive created, application submitted, offer accepted. Use a centralized track(event, props) utility so the analytics provider can be swapped.
No Session Replay Global
All pages		 No session replay tool (Hotjar, FullStory, LogRocket, PostHog Session Replay) is integrated. Without session replay, reproducing user-reported bugs requires guesswork. PII-sensitive fields (passwords, CGPA, salary) would need to be masked in replay config — but since no replay is configured, this is not yet a risk. Medium If integrating PostHog for analytics (above), enable Session Replay with automatic PII masking: mask_all_text: false but add data-ph-no-capture to salary, CGPA, password, and email fields. This gives bug reproduction capability without exposing sensitive data in recordings.
No Frustration Signals Global
Forms, tables, actions		 No tracking of frustration signals: dead clicks (clicking non-interactive elements), rage clicks (5+ rapid clicks), excessive form field edits (editing the same field 3+ times), or form abandonment. These signals identify UX friction that users don't report verbally. Medium PostHog and Hotjar both capture rage clicks automatically. For dead clicks, add a click event listener that checks if the click target has a registered handler — log events where it doesn't. For form abandonment, track beforeunload events when formDirty === true to identify forms users fill partially then leave.

6 · Security UX Category 64 issues

The two highest-severity security UX issues (password field type="text" and raw result.error surfacing) were documented in the prior audit. This audit finds two additional gaps: password requirements are shown only on validation failure (post-submit), not proactively — and no sensitive action requires re-authentication.
Sub-CriterionScreen / FileIssueSeverityFix
Password Policy Display Register Page
register/page.js		 Password requirements ("must be at least 8 characters with uppercase, lowercase, and number") are defined in validators.js but shown only when validation fails after submission. A new user types a weak password, clicks submit, fails, reads the requirements for the first time, then types a new password. This is friction-first UX — requirements should be visible before the user starts typing. High Show password requirements below the password field by default as a checklist: ✓ 8+ characters · ✓ Uppercase · ✓ Lowercase · ✓ Number. Each criterion turns green as the user types and satisfies it. This is a live validation pattern — use onChange to test each rule individually and update the UI. Remove the post-submit error for password strength once this is in place.
Session Expiry UX Global
All authenticated pages		 When the 24-hour session expires, API calls return 401. Since most pages lack error handling for failed SWR fetches, the skeleton renders indefinitely. No countdown warning is shown before expiry. A college admin filling a 15-field drive creation form for 25 minutes will lose all their work to a silent session expiry with no warning. High Use NextAuth's useSession to compute time-to-expiry. Show a fixed banner at 10 minutes remaining: "Session expires in 10 minutes — Extend session". On API 401 response in the SWR fetcher, redirect to login with ?reason=session_expired — show an explanatory message on the login page. Auto-save form state to sessionStorage before redirect; restore it on login return.
Sensitive Action Re-auth Admin pages
Delete employer, bulk operations		 High-consequence actions (revoking employer campus access, bulk-deleting students, changing admin permissions) require no re-authentication. Once logged in, any college admin can perform any admin action without additional identity verification. On a shared computer (college lab), this is a significant risk. Medium For irreversible admin actions, add a re-authentication step: prompt for password before executing — the same pattern used by GitHub/Google for account deletion. Use a modal: "This action is irreversible. Enter your password to confirm." Verify server-side by calling a dedicated /api/auth/verify-password endpoint.
Mixed Trust Signals EntityLogo component
EntityLogo.js · Clearbit fetch		 The EntityLogo component fetches company logos from Clearbit's public API (https://logo.clearbit.com/{domain}). This loads third-party image content in the browser without user consent or visibility. No visual indicator distinguishes Clearbit-loaded logos from internally hosted assets. If Clearbit serves a redirect to a malicious URL, the browser will follow it silently. Low Add referrerPolicy="no-referrer" to the Clearbit <img> element to prevent leaking the app URL to the third party. Configure Next.js next.config.js to allowlist logo.clearbit.com in images.remotePatterns — this routes the request through Next.js Image optimization, adding a server-side proxy that prevents direct browser-to-Clearbit contact.

7.1 · Visual & UI Systems Category 74 issues

The CSS custom property architecture in globals.css is comprehensive — semantic color layers, spacing scale, radius scale, shadow elevation, and transition tokens are all defined. The gap is at the usage layer: 23–32 inline style={{}} objects per page bypass the token system, and the emoji icon set is the single largest "template tell" preventing enterprise-grade visual quality.
Sub-CriterionScreen / FileIssueSeverityFix
Design Debt Density Dashboard overview pages
student, employer, college		 Random sample of three overview pages finds 23–32 inline style={{}} objects per file. Inline styles cannot be overridden by theme switches or media queries without !important, are invisible to linters, duplicate values across files, and represent the largest single source of design inconsistency. Total estimated inline styles across 55 pages: 1,200–1,500 individual style declarations. High Define a rule: inline styles only for dynamic computed values (e.g., progress bar width from data). Static spacing, colors, and font sizes become CSS utility classes. Add to globals.css: .mt-6, .mb-4, .gap-3, .text-sm, .font-semibold, .text-secondary. A migration script can identify static inline styles: grep -r "style={{ [a-zA-Z]" src/ | grep -v "width.*%" | wc -l.
Emoji Icon System Sidebar, stats cards, page headers
layout.js, dashboard pages		 The sidebar uses 30+ emoji as navigation icons. Stats cards use emoji at font-size: 1.5rem. Emoji render at inconsistent sizes across OSes (Windows vs macOS vs Android), do not respect CSS color inheritance (cannot be tinted for active/dark mode states), and are announced verbatim by screen readers. This is the dominant "template" signal in the UI that prevents enterprise-grade perception. High Lucide React is already partially in use (PageError, forms). Extend to sidebar and stats cards. Replace all sidebar emoji in menuConfig with Lucide icon components: icon: <LayoutDashboard size={18} aria-hidden="true" />. For stats cards, use FileText, CheckCircle, Calendar, Gift. Lucide icons inherit currentColor — dark mode, active state, and hover tinting work automatically.
Spatial Density Mixing College students page
Employer jobs page with form open		 When the inline create form opens on employer/jobs, the compact data-table density (12 columns, tight rows) and the relaxed form density (large inputs, generous padding) are rendered simultaneously in the same viewport. The density contrast is jarring and creates an unbalanced visual weight distribution — the form feels like it belongs to a different product. Medium When the create form opens, slide the table into a collapsed/reduced view (show only 3 key columns, reduce row height by 30%). Alternatively, use a side panel pattern: the form slides in from the right, narrowing the table to 60% width. This maintains context (user can see the existing list) while separating the density contexts. The single modal-overlay in drives shows this is already achievable.
Typography Rhythm All dashboard pages
Non-scale font sizes		 Non-scale font sizes (0.9375rem, 0.8125rem, 0.6875rem, 0.68rem) appear across multiple pages. These do not align to any mathematical scale (neither 1.125 Major Second nor 1.25 Major Third). The inconsistency causes subtle vertical rhythm breaks — adjacent text blocks have mismatched line spacing that creates an unpolished finish on close inspection. Medium Enforce a 5-step scale: 0.75rem (xs) · 0.875rem (sm) · 1rem (base) · 1.125rem (lg) · 1.25rem (xl). Add these as CSS utility classes. Audit all inline font-sizes: any value outside this scale is replaced with the nearest step. The sub-10px value 0.6rem in the topbar is eliminated — replace the text-based chevron with an SVG icon.

7.2 · Interaction & Micro-Behavior (Deep) Category 73 issues

The scale(0.97) button press and translateY(-2px) card hover are genuine polish details. The loading state hierarchy is incomplete — only skeleton and button-level spinners exist; no blocking overlay for multi-step async flows. The notification panel has no Escape/outside-click dismissal (addressed in section 2 but confirmed here as a micro-behavior gap too).
Sub-CriterionScreen / FileIssueSeverityFix
Loading State Hierarchy Multi-step forms
File upload (student/documents)		 Only two loading states exist: skeleton (full-page) and spinner-in-button (inline). No blocking overlay exists for flows where the user must not interact during processing — e.g., file upload to S3 (student documents), bulk status updates (once implemented), or multi-step API chains. The drives page has one modal-overlay class defined — but it is not used for loading states. High Add a third loading tier: <FullPageOverlay /> — a semi-transparent backdrop with a centered spinner and message like "Uploading document..." or "Saving changes...". Use it for: file uploads (documents), any operation taking >1 second. The existing modal-overlay CSS class is the right foundation — wrap it in a component with a message prop.
Keyboard Interaction Cues All interactive elements
globals.css		 No keyboard shortcut hints are displayed anywhere in the UI. Even simple shortcuts (Tab to navigate, Enter to submit, Escape to close) are not surfaced as affordances. Power users have no discovery path for any keyboard behaviour that does exist. There is no ? shortcut panel or tooltip hint pattern. Medium Add title attributes to key actions with their keyboard equivalent: title="Submit (Enter)" on primary form buttons, title="Close (Esc)" on panel close buttons. Add a ? floating button that opens a keyboard shortcuts reference modal. Long-term: show shortcut hints inside tooltips: [G then S] Students.
Micro-copy Timing Form fields
All helper texts		 Helper text and field hints appear immediately on mount alongside labels — not on focus/interaction. Critical guidance (e.g., "CTC range must overlap minimum wage — format: ₹500,000") is shown to all users at all times, creating visual clutter for experienced users. Supplementary info should appear at different speeds than critical warnings. Low Show field-level hints only when the field is focused: const [focused, setFocused] = useState(false). For critical validation feedback, show immediately on blur (<200ms). For supplementary tips ("You can add multiple branches separated by comma"), show on a 500ms delay after focus — so fast typists never see them, but pausing users do.

7.3 · Cognitive & Behavioral UX Category 73 issues

The primary action hierarchy is generally clear — primary buttons are visually dominant, secondary actions use ghost variants. The main cognitive UX gap is the employer/jobs page with the form open: 13+ active form inputs, 4 filter tabs, 3 campus checkboxes, and the existing jobs table are all visible simultaneously — exceeding the 7–9 interactive element threshold for a single viewport.
Sub-CriterionScreen / FileIssueSeverityFix
Cognitive Load Employer Jobs — form open
employer/jobs/page.js		 When the create-job form is open alongside the jobs list, the viewport contains: 1 page header button, 1 form close button, 13 form inputs/selects, 2 submit buttons (Draft/Publish), 4 filter tabs, and the existing jobs table with action buttons per row. This is 20+ interactive elements in simultaneous view — 2–3× the cognitive load threshold. First-time employers are overwhelmed. High Move job creation to a dedicated /dashboard/employer/jobs/new page, or use a full-width slide-over panel that replaces the table view entirely when open. Collapse the 12-field form into 2 steps: Step 1 — basics (title, type, salary, vacancies), Step 2 — targeting (campuses, CGPA, deadline). This reduces per-view interactive element count to under 8.
Progressive Disclosure All filter toolbars
Employer jobs, college students		 All filter options are shown simultaneously with no distinction between "common" and "advanced" filters. The college students page shows status filter, department filter, and search at the same level with equal visual weight. Power users need all of them; new users are confused by options they don't yet understand. Medium Show only 1–2 primary filters by default (status + search). Add an "More filters" button that expands to reveal advanced filters (department, CGPA range, placement status, year). Use smooth CSS height animation for the expansion. Remember expanded state in localStorage so power users don't need to expand every time.
Empty/Zero/Null Distinction All pages with data tables
students, drives, applications		 All three empty states use the same .empty-state component with no variation: "No data yet" for a new college with no students (empty), "No results" for a search that matches nothing (zero), and the same pattern for API errors returning no data (null). The user cannot tell whether to take action (add a student) or change their filter (clear search) or contact support (data error). Medium Distinguish three patterns: Empty (new account, no records) — show icon + encouraging message + primary CTA ("Add your first student"). Zero (search/filter returned nothing) — show "No results for [query]" + "Clear filters" button. Null (data error or unexpected absence) — show warning icon + "Contact support" link. Each gets a distinct icon and tone.

7.4 · Edge Cases & State Design Category 72 issues

Partial data and system failure specificity are the two gaps here. The EntityLogo component handles missing images elegantly (layered fallback: URL → Clearbit → gradient+initials). The failure is in API error messages — all errors say "Failed to load data" regardless of type — and in permission-based UI variation, which uses complete hiding rather than the more informative disabled-with-explanation pattern.
Sub-CriterionScreen / FileIssueSeverityFix
Failure Specificity PageError component
components/PageError.js		 The PageError component shows "Failed to load data" with the raw error.message or "There was an unexpected issue retrieving this information." It does not differentiate: network timeout (user's connection issue), server 500 (backend bug), 401 (session expired), 403 (permission denied), or 404 (entity deleted). Each case needs a different response from the user. High In the SWR fetcher, attach an HTTP status code to the error: const err = new Error(json.message); err.status = res.status; throw err. In PageError, switch on error.status: 401 → "Session expired — Sign in again"; 403 → "You don't have permission to view this"; 404 → "This page no longer exists"; 0/network → "Check your connection + Retry"; 500 → "Something went wrong on our end + Contact support".
First-Time vs Returning All dashboards
All 4 role dashboards		 No distinction is made between a first-time user (no data, needs guidance) and a returning power user (has data, needs efficiency). A new student sees the same empty dashboard layout as an experienced student with 10 applications — just with empty cards. A new employer with no jobs posted sees the same interface as one with 50 active postings. The UI does not adapt. Medium Add a computed isNewUser flag to the dashboard API response: true if the user has been registered less than 7 days and has fewer than 3 actions. For new users, show the onboarding checklist card (from SaaS audit, issue #16) instead of the standard stats grid. For returning users, show the stats grid. No UI-level logic beyond a conditional card render.

7.5 · Trust, Perception & Product Feel Category 72 issues

This is the strongest category. The card system, shadow elevation, gradient stats cards, EntityLogo layered fallback, and dark mode implementation all signal a genuinely polished product. No dark patterns were found anywhere in the codebase. The primary gap is an absence of delight moments and an overuse of native confirm() breaking the friction-vs-reassurance balance for destructive actions.
Sub-CriterionScreen / FileIssueSeverityFix
Friction/Reassurance Balance Destructive actions
3 pages using confirm()		 Routine actions (submit form, publish job) have appropriate one-click friction. Destructive actions (block employer, cancel booking, delete document) incorrectly use window.confirm() — a browser-native dialog with no context, no entity name, and no consequence description. The friction level is wrong: too abrupt for employer revocation, which affects an entire institution, and too harsh for document removal, which should be an undo. High A ConfirmDialog component already exists in /components/ConfirmDialog.js but is not used for any of the 3 confirm() instances. Wire it up: for employer block, show entity name + consequence; for booking cancel, show a lighter warning + undo option; for document delete, implement soft-delete + undo toast instead of a warning dialog entirely.
Delight Moments Key success flows
Offer accepted, first application		 No delight moments exist. When a student submits their first application, a major milestone, the feedback is a standard success toast. When an offer is accepted — one of the most emotionally significant events in a placement platform — the UI shows no celebration. The EntityLogo gradient is the only designed-in "pleasant surprise"; it doesn't count as a delight moment. Low Add targeted micro-celebrations: (1) First application submitted → confetti burst for 2 seconds using a CSS keyframe animation (no library needed). (2) Offer accepted → full-screen momentary overlay with a congratulations message + the company logo. (3) All placement drives completed → college dashboard shows a summary celebration card. These moments are remembered and shared — high brand impact, low implementation cost.

7.6 · Scalability & System Thinking Category 73 issues

The CSS grid system uses minmax() flexibly in several places — a solid foundation. The button and badge component systems are well-defined and consistent across pages. The primary scalability risks are the inline style density (1,200+ inline declarations estimated, documented in 7.1), the absence of design system documentation, and several fixed-pixel grid columns that will break with dynamic content.
Sub-CriterionScreen / FileIssueSeverityFix
Fixed Grid Columns globals.css
Table grid layouts		 Several grid layouts use fixed pixel column widths: grid-template-columns: 140px 1fr 150px, 160px 1fr 200px. These will break when: (a) translated to a language with longer strings, (b) a new column is added by a developer, or (c) the viewport is narrower than expected. At least 4 such fixed-column definitions exist in globals.css. Medium Replace fixed-pixel columns with minmax(): grid-template-columns: minmax(120px, 160px) 1fr minmax(130px, 200px). This gives a minimum floor (prevents collapse) and a maximum ceiling (prevents overflow) while allowing intermediate flexibility. For label-value grids, use auto 1fr which is both flexible and label-length-adaptive.
No Design System Docs globals.css
Component library		 The CSS custom property system in globals.css is the implicit design system, but it has no documentation. A new developer joining the team has no reference for: which tokens to use for which contexts, which spacing step to use for card padding vs section margins, which shadow elevation to apply to modals vs cards, or which badge variant to use for which status. This creates drift over time. Medium Create a /design-system route (accessible in development only) that renders all tokens: color palette, spacing scale, shadow elevations, border radii, typography scale, button variants, badge variants, and empty state patterns. This is a one-page living reference — no Storybook required. A junior developer can see all available components and tokens in one place.
Long Content Resilience Student name display
Company name display, job titles		 Company names (employer profile, job cards, drive cards) and student names are not protected against overflow. A company named "International Business Machines Corporation" or a student named "Thiruvananthapuram Krishnamurthy Subramanian" will overflow card boundaries or force text wrapping that breaks card layout. No text-overflow: ellipsis or max-width is defined for these fields in the card system. Low Add to the card system CSS: .card-entity-name { overflow: hidden; text-overflow: ellipsis; white-space: nowrap; max-width: 100%; }. For multi-line contexts (descriptions), use display: -webkit-box; -webkit-line-clamp: 2; overflow: hidden. Apply to: company name in drive cards, job title in job cards, student name in applications table. Add title={fullName} so the full value is accessible on hover.

What Is Working Well

AreaStrength Observed
Dark Mode Implementation Dark mode is user-controlled (not system-only), persisted to localStorage, and applied via a CSS attribute selector on <html data-theme="dark">. The token system fully inverts — background, text, border, shadow, and badge colors all have explicit dark overrides. This is implemented better than most production SaaS applications.
EntityLogo Fallback Chain The layered fallback (explicit URL → Clearbit API → deterministic gradient + initials) is elegant. The imgLoaded state pattern (gradient always rendered underneath, image shown on onLoad) prevents any broken-image state entirely. The gradient is seeded from the entity name, so the same company always gets the same color — a subtle but meaningful consistency.
Button Micro-interactions The button system has genuine tactile quality: transform: scale(0.97) on active state creates a press feel, translateY(-2px) on card hover creates lift, and gradient backgrounds with inner-highlight (inset 0 1px 0 rgba(255,255,255,0.1)) add depth. These details are correct and well-implemented.
No Dark Patterns A full audit found zero dark patterns: no hidden auto-renewals, no obscured cancellation paths, no fake scarcity signals, no confirm-shaming language, and no manipulative UI. The application is honest and straightforward in its UX — a meaningful baseline for trust.
Token Architecture The CSS custom property system in globals.css is comprehensive: semantic color layers, a complete 5-step spacing scale, border radius scale, shadow elevation set (5 levels), and transition tokens. The naming is semantic (--text-secondary not --gray-400). The foundation is correct — the failure is at the usage layer (inline styles bypassing it).
Grid Flexibility The CSS grid system uses minmax() in key layouts (sidebar width, card grids, responsive breakpoints). The card grid collapses from 3-column to 2-column to 1-column at defined breakpoints. The layout is adaptive in most contexts — the fixed-pixel columns noted above are exceptions, not the rule.
