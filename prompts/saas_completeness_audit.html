Edge Cases
API Failure Handling
 29 of 36 SWR-fetching pages have no error state. When the API returns 500 or the network fails, the skeleton renders indefinitely. Users cannot tell if data is loading or broken, and there is no retry path. Only 7 pages import PageError. High Create a shared <PageError onRetry={() => mutate()} /> pattern and apply it to all remaining 29 SWR pages. Add it to the SWR guard: if (error) return <PageError error={error} onRetry={mutate} /> before the loading check. YES P7
2
Data Tables
Pagination
 All API routes use hardcoded LIMIT 500 or LIMIT 1000. At real scale (college with 5,000 students, employer with 2,000 applications), these queries will time out or return unmanageable payloads. The one pagination UI in employer/requests shows a toast "Pagination is not implemented yet" — confirming this is a known gap. High Add ?page=1&pageSize=50 query params to API routes. Return { data, total, page, pageSize }. Build a reusable <Pagination> component. Priority tables: students, applications, jobs, drives. YES P1
3
CRUD
Soft Delete
 No entity in the schema has a deleted_at or is_deleted column. All deletes are permanent. In a placement platform, deleting a drive, offer, or student profile is legally and operationally irreversible — audit compliance requires recoverable records. High Add deleted_at TIMESTAMPTZ to key tables: placement_drives, offers, jobs, student_profiles. Change DELETE routes to UPDATE … SET deleted_at = NOW(). Add WHERE deleted_at IS NULL to all GET queries. YES P2
4
Automation
Reminders — Hardcoded
 The student/reminders/page.js exports a hardcoded const REMINDERS = [...] array (TechCorp, Infosys, DataVerse). It is the only remaining page with dummy data. The page header says "Preview inbox — no email is sent from this screen" — but it shows static company names to every student regardless of their actual drives. High Wire to /api/student/drives and filter drives within the next 7 days to generate real reminders. Remove const REMINDERS array. Show actual upcoming drives from the student's college calendar. YES P5

Messaging
Destructive Warnings
 Three destructive actions use native window.confirm(): "Block this employer" (college/employers), "Cancel this booking" (college/infrastructure), "Remove this document" (student/documents). The browser dialog has no context, is unstylable, is blocked by some enterprise browsers, and breaks the app's visual language. High Build a reusable <ConfirmDialog title message confirmLabel onConfirm /> modal. Replace all 3 confirm() calls. The modal can display the entity name, the consequence ("This employer will lose campus access"), and require an explicit "Block" button click. YES P9

Bulk Actions
 Zero bulk operations exist anywhere in the application. A college admin managing 500 students cannot bulk-update placement status. An employer reviewing 200 applications cannot bulk-shortlist or bulk-reject. This is the single biggest productivity gap for users operating at real scale. High Add checkbox column to key tables (students, applications). Show a bulk action toolbar when ≥1 row is selected. Priority actions: bulk update placement status (students), bulk shortlist / bulk reject (applications), bulk publish (jobs). YES P8

Compliance
GDPR / Data Export
 No mechanism exists for a student to export or delete their personal data. The audit_logs table exists in the schema but there is no UI to view it. For a platform storing PII (names, emails, CGPA, documents), the absence of a "Download my data" and "Delete my account" flow is a compliance risk. High Add a "Privacy" section to student/profile: "Download my data" (generates JSON of all student rows) and "Request account deletion" (sets a deletion_requested flag, sends to admin review). Add an admin UI to view audit_logs by user/tenant. YES P14
8
Data Tables
Sorting
 No table in the application has clickable column headers for sorting. Data renders in database insertion order. A college admin looking for the highest-CGPA student, or an employer finding the most-recent application, has no way to sort without exporting to CSV first. High Add a sort icon to column headers. Store [sortCol, sortDir] in state. For client-side tables, sort the in-memory array. For server-side tables, append ?sortBy=cgpa&sortDir=desc to the API call. Default: created_at DESC for all tables. No P1

Search UX
No Debouncing
 All search fields fire filter() on every keystroke synchronously. The debounce utility function exists in src/lib/utils.js but is not used in any search field. For client-side filtering this causes unnecessary renders on large datasets; for any future server-side search this will cause a request flood. High Wrap all search onChange handlers with the existing debounce(fn, 300) utility. Alternatively, add a useDebounce(value, 300) hook and apply it to the search state variable before filtering. 5-minute fix per page. No P11

CRUD
Native confirm() dialogs
 Three pages use confirm()/window.confirm() for destructive action warnings. These block the browser main thread, cannot be styled, are interceptable by browser extensions, and in some enterprise browser policies are suppressed entirely — meaning the delete executes without any warning. High Replace all 3 instances with a reusable confirmation modal. Pages affected: college/employers (block employer), college/infrastructure (cancel booking), student/documents (remove document). No P2

Search UX
Typo Tolerance
 All search is exact string match via JavaScript includes(). Typing "infosys" finds "Infosys" (case-insensitive), but "infisy" or "infossy" returns zero results. In a mobile context where students type company names quickly, misspelling by one character returns nothing. Low For client-side search, integrate a lightweight fuzzy-match library (e.g. fuse.js — 4KB). Configure with threshold 0.4. Replace .filter(item => item.name.includes(q)) with fuse.search(q).
