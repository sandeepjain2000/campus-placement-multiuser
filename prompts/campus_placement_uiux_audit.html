1 · Design Token Enforcement 5 issues

The design token foundation in globals.css is well-structured — Inter font, semantic color layers, spacing scale, radius scale, and transition tokens are all defined. The gap is at the usage layer: page components bypass the token system with hardcoded hex values and arbitrary rem fractions, undermining the system's value.
Category Screen / Component Issue Severity v0-Standard Fix
Token Bypass Login Page
login/page.js · success banner		 Success info banner after clicking a demo card uses hardcoded hex: #f0fdf4, #bbf7d0, #166534. These exact values already exist as var(--success-50), var(--success-100), var(--success-700). In dark mode these inline colors will not adapt. High Replace all three hardcoded hex values with their token equivalents: background: var(--success-50), border-color: var(--success-100), color: var(--success-700). Extract the banner into a reusable <InlineAlert variant="success"> component that accepts info | success | warning | danger and maps to the correct token pair automatically.
Token Bypass Button System
globals.css · .btn-danger:hover		 The danger button hover state uses #BE123C hardcoded — this is Tailwind's rose-700 but is not defined as a token. The project defines --danger-50 through --danger-600 only; --danger-700 is missing, leaving the developer to fall back to a raw hex. High Add --danger-700: #BE123C to the :root token block and update the hover rule to var(--danger-700). While there, add the full missing ramp: --danger-700, --success-700 already exists but confirm --info-700 and --warning-700 are also present for consistency.
Missing Tokens Dark Mode Badges
globals.css · [data-theme='dark'] .badge-*		 Dark mode badge overrides use hardcoded rgba() values (e.g., rgba(16,185,129,0.15)) instead of extending the semantic token system. Any future change to the success color requires updating both the token and the raw rgba — the two are silently disconnected. Medium Add dark-mode token overrides to the [data-theme='dark'] block: e.g. --success-100: rgba(16,185,129,0.15) (dark variant). Badge CSS then uses var(--success-100) unconditionally — the theme switch handles both modes. This collapses ~10 selector overrides into token values.
Arbitrary Values EntityLogo Component
EntityLogo.js · sizeMap		 Logo sizes are hardcoded integer pixels: xs=20, sm=32, md=40, lg=56, xl=80. These live outside the token system. If a designer changes the spacing scale, logos will not respond. The border-radius values (4px, 6px, 8px, 10px, 14px) are also raw numbers rather than token references. Medium Map sizes to spacing tokens: xs: var(--space-sm) → 8px is too small, but the pattern should use CSS custom properties exposed from the component: --entity-logo-size: 40px as a component-scoped variable. Border radius should map to var(--radius-sm) through var(--radius-xl) per size tier.
Incomplete Scale Topbar · Sidebar
dashboard/layout.js		 The topbar identity section contains 6 inline fontSize values: '1rem', '0.75rem', '0.875rem', '0.6rem', '0.8125rem'. While most map loosely to the h-scale, '0.6rem' (used for the campus selector dropdown arrow) falls below the smallest defined step and is not a usable readable size. Low Define all text sizes used in layout as semantic role classes: .topbar-entity-name, .topbar-entity-role, .topbar-section-label. Move font-size into globals.css. Remove the sub-10px '0.6rem' entirely — use a proper chevron SVG icon instead of a Unicode arrow character.

2 · Typography Scale & Hierarchy 4 issues

Inter is loaded correctly and the h1–h6 element scale is defined. The problem is that no component uses the defined heading elements — all headings are <div> or <h4> tags with inline fontSize overrides using non-scale values like 0.9375rem and 0.6875rem. The result is a brittle typography hierarchy that varies arbitrarily between screens.
Category Screen / Component Issue Severity v0-Standard Fix
Arbitrary Sizes Student Dashboard
student/page.js		 Four non-scale font sizes within a single page: 0.9375rem (card title), 0.8125rem (drive metadata row), 0.6875rem (drive type badge), 0.68rem (demo login card). None of these exist in the defined scale (0.75, 0.875, 1, 1.125, 1.25…). They are Tailwind pixel-step equivalents hand-translated to rem without a scale anchor. High Align all sizes to the defined scale: 0.9375rem → 1rem (text-base), 0.8125rem → 0.875rem (text-sm), 0.6875rem → 0.75rem (text-xs), drop 0.68rem entirely and use text-xs. Add CSS utility classes to globals.css: .text-xs { font-size: 0.75rem } through .text-4xl so pages use class names, not inline values.
Heading Misuse All Dashboard Pages
page-level <h1> in .page-header		 The .page-header pattern renders a <h1> for every page section title. When a page has a card with its own .card-title (rendered as a <h3>), the DOM has both an h1 (page title) and multiple h3 (card titles) but no h2 — skipping a heading level. Screen readers use heading hierarchy for navigation; skipped levels confuse the outline. Medium Establish a clear heading hierarchy: h1 = page title (one per page), h2 = section/card group, h3 = individual card title, h4 = sub-section. Update .card-title from <h3> to the correct level based on context. Never skip levels. Visual size is controlled via CSS — semantic level is controlled via the correct element.
Weight Inconsistency Sidebar Navigation
dashboard/layout.js		 Sidebar section titles (e.g., "Overview", "Placements", "Management") use .sidebar-section-title but that class is not present in the visible CSS; section titles and nav link labels may render at the same visual weight, making grouping invisible unless colour alone distinguishes them. Medium Define .sidebar-section-title explicitly: font-size: 0.6875rem; font-weight: 700; text-transform: uppercase; letter-spacing: 0.08em; color: var(--text-tertiary); padding: 1rem 1rem 0.375rem. Nav link labels should be font-size: 0.875rem; font-weight: 500. This 3-level hierarchy (section title → link → sub-label) creates scannable groups.
Line Height Login Demo Cards
login/page.js · email/password lines		 Demo card email/password lines are styled at lineHeight: 1.6 and 0.68rem font-size. At sub-12px sizes, the line-height appears too generous and the text looks lost. The adjacent name line uses no explicit line-height, making the two-line block feel unrelated. Low Use font-size: 0.75rem and line-height: 1.4 for the credential lines. Group email and password in a flex flex-col gap-0.5 container with a top margin separating it from the name line. This creates clear visual grouping within each card.

3 · Interaction & Micro-Behavior 5 issues

The button press scale (transform: scale(0.97)), card hover lift (translateY(-2px)), and spring transition token are defined and working well. The gaps are in feedback completeness — loading states collapse button widths, focus rings are invisible on buttons, and the single use of alert() for a module-unavailable state breaks the polished illusion entirely.
Category Screen / Component Issue Severity v0-Standard Fix
Native Alert Login Page
login/page.js · handleDemoCardClick		 Clicking the "Placement Committee" demo card triggers alert('The Placement Committee module…'). This native browser dialog freezes the tab, ignores the app's visual theme, cannot be styled, and is blocked by many corporate browsers. It is the single highest-friction interaction on the entire platform. High Replace with an inline toast notification using the existing ToastProvider: show a variant="info" toast — "Placement Committee is coming soon. It's currently in design." — that auto-dismisses after 5 seconds. Optionally, show the card in a visually distinct "coming soon" state (greyed badge + lock icon) so users know before clicking.
Button Loading Login · All Forms
login/page.js · submit button		 The submit button loading state changes text to "Signing in…" but: (1) there is no spinner — users with slow connections get no progress indication, (2) the button width collapses when the shorter text replaces "Sign In →", causing layout jump, (3) no aria-busy attribute is set — screen readers do not announce that an action is in progress. High Add min-width matching the default label width to the button. Replace the text with an inline spinner + "Signing in…": <span aria-hidden="true" className="spinner-sm" /> Signing in…. Add aria-busy="true" to the button during loading. Define .spinner-sm in globals.css: 16px circle with border animation, matching the button's text color.
Focus Ring All Interactive Elements
globals.css · .btn		 The .btn class defines :active { transform: scale(0.97) } but has no :focus-visible rule. Buttons receive no visible focus ring when navigated by keyboard. Form inputs have a glow (box-shadow: 0 0 0 3px rgba(79,70,229,0.12)) but this is an inner glow, not a proper offset outline, and it fails at low contrast in dark mode. High Add to globals.css: .btn:focus-visible { outline: 2px solid var(--border-focus); outline-offset: 2px; }. For inputs, replace the box-shadow with: .form-input:focus-visible { outline: 2px solid var(--border-focus); outline-offset: 0; box-shadow: 0 0 0 4px rgba(79,70,229,0.12); }. The outline provides the WCAG-required 3:1 ratio; the box-shadow adds the glow purely for aesthetics.
Reduced Motion Global
globals.css · all transitions		 prefers-reduced-motion is not implemented anywhere in globals.css. All hover transforms (translateY(-2px), translateY(-3px)), scale effects, skeleton shimmer animations, and entrance animations (animate-fadeIn, animate-slideUp) will fire for users who have set "reduce motion" in their OS accessibility preferences — triggering vestibular symptoms in sensitive users. Medium Add a single block to globals.css: @media (prefers-reduced-motion: reduce) { *, *::before, *::after { animation-duration: 0.01ms !important; transition-duration: 0.01ms !important; scroll-behavior: auto !important; } }. For skeleton shimmer specifically, replace animation with a static background: the shape is still informative without motion. For page entrances, replace opacity+transform with opacity-only at duration 50ms.
Notification Panel Topbar
dashboard/layout.js · notif dropdown		 The notification panel opens on bell click but has no close affordance (no × button), no click-outside-to-close handler, and no Escape key handler. Once open, it can only be closed by clicking the bell again — which is discoverable only by trial. In mobile view the panel likely clips outside the viewport with no repositioning logic. Medium Add a close button (×) to the panel header. Add a useEffect with a mousedown outside-click handler to close the panel. Add onKeyDown on the bell button to close on Escape. For mobile: apply position: fixed; right: 0; left: 0; top: var(--topbar-height) on screens below 640px. Add role="dialog" and aria-label="Notifications".

4 · Form Design & Input States 4 issues

The form system in globals.css defines .form-input, .form-label, .form-error, and .form-hint — a solid foundation. The failures are in how these patterns are applied: the most critical form on the entire platform (login) uses type="text" for the password field, and inline styles for error display rather than the defined .form-error class.
Category Screen / Component Issue Severity v0-Standard Fix
Password Field Login Page
login/page.js · password input		 The password field uses type="text" — the comment says "so demo passwords are visible." This means every user's password is visible in plain text in the browser and in any shoulder-surfing scenario. Password managers will not offer to save the field. The browser will not trigger autocomplete for the correct credential. It also trains users to accept visible passwords. High Use type="password" unconditionally. Add a show/hide toggle button (eye icon) inside the input, toggling between type="password" and type="text" on click — with aria-label="Show password" / "Hide password". For the demo experience, the "Credentials filled" banner already confirms the account is selected — users can click Show to verify if needed.
Error Display Login Page
login/page.js · error block		 Login error is shown as a <div> with inline style={{ padding: '0.75rem 1rem', background: 'var(--danger-50)', ... }}. The project already defines .form-error in globals.css. Using inline styles here means: dark mode won't adapt the error banner background (it uses the token but in an inline context that's fine, however the pattern is inconsistent), and the error cannot be composed with icon + dismiss. Medium Create a reusable <FormAlert variant="danger|success|warning|info"> component with an icon (SVG, not emoji), message text, and optional dismiss button. Use it for both the error state and the "credentials filled" success state. Style in globals.css under .form-alert with variant modifiers. This unifies error display across all 41 pages.
Demo Credentials Login Page
login/page.js · role-select-grid		 Every demo account card displays the raw password "Admin@123" as visible text via 🔑 Admin@123. While this is a demo app, shipping this pattern to staging/production exposes credentials on the public-facing login page. It also establishes an anti-pattern: users learn that passwords are meant to be shown in the UI. Medium Replace the password display with a "Click to auto-fill" instruction pattern. Remove the password from the card entirely — use a single line: "Auto-fills credentials". On click, fill both fields and show the success banner. If the demo password must be shown for development, gate it behind a process.env.NEXT_PUBLIC_SHOW_DEMO_PW === 'true' environment variable.
Controlled Input College Admin Topbar
dashboard/layout.js · session select		 The academic session <select> uses defaultValue="2025-26" (uncontrolled) instead of value + onChange (controlled). In React, mixing controlled and uncontrolled patterns on the same element causes unpredictable behaviour — the selection will not persist through renders, will not reflect in state, and cannot be used to filter data downstream. Medium Lift the session selection into a state variable (or URL query param for persistence): const [session, setSession] = useState('2025-26'). Pass value={session} onChange={(e) => setSession(e.target.value)}. Persist the selected session to localStorage so it survives page navigation. Propagate the selected year to all data fetching calls via context or URL param.

5 · State Design (Loading · Empty · Error) 4 issues

Skeleton loading screens and the three-tier empty state pattern are partially implemented — a genuine strength. The gap is that error states from failed data fetches are completely unhandled: if SWR's error flag is truthy, the skeleton renders indefinitely. Hardcoded data in the notification panel and stats cards also creates false confidence in the live data experience.
Category Screen / Component Issue Severity v0-Standard Fix
Error State All SWR Pages
student/page.js + 40 other pages		 Every SWR-fetching page uses if (isLoading || !data) { return skeleton }. When SWR's error is truthy (network failure, 500 response, auth expiry), the condition !data is also true — the skeleton renders indefinitely with no error feedback. There is no way for users to know the fetch failed or to retry. High Split the loading guard: if (error) return <PageError onRetry={() => mutate()} /> first, then if (isLoading || !data) return <Skeleton />. Create a shared <PageError> component showing the specific error type: network failure shows "Check your connection + Retry", auth error redirects to login, server errors show "Something went wrong + Contact support". Add this to all 41 data-fetching pages.
Hardcoded Data Student Dashboard
student/page.js · stats-card-change		 The stats card change indicator displays "↑ 2 this week" as a static string in the JSX. This is the same hardcoded string for every user regardless of their actual application activity. Users with 0 applications will see "↑ 2 this week" — eroding trust in the data immediately. High Either: (a) add a weeklyChange field to the API response for each stat and render it conditionally — only show the change badge if the value is non-zero, or (b) remove the change indicator entirely until the API supports it. Never show fabricated trend data. If the API change is deferred, replace with a neutral label like "This semester".
Hardcoded Data Topbar Notifications
dashboard/layout.js · notification panel		 All three notification items are hardcoded JSX strings ("Google Campus Drive Approved", "New Application Received", "Offer Deadline Approaching"). They are identical for every user, every role, every session. The notification dot always appears. Users who receive real notifications via the /dashboard/alerts page will find the topbar panel is decorative. Medium Connect the notification panel to /api/notifications (or the same endpoint used by the alerts page) via SWR with a short refreshInterval: 60000. Show maximum 3 most recent unread notifications. Show the dot badge only when unreadCount > 0. Show an empty state ("You're all caught up ✓") when none. Add a loading skeleton for the panel content.
Skeleton Fidelity All Dashboard Pages
loading fallback pattern		 The dashboard loading fallback renders generic skeleton skeleton-heading and skeleton skeleton-card classes. The stats cards are 4-column grids with icon, value, label, and change badge — but the skeleton likely renders a flat rectangle at card dimensions, causing a significant layout shift when real content replaces it (different heights for icon block vs flat rect). Low Build skeleton variants that mirror the actual component structure: the stats card skeleton should have a skeleton block for the icon area (48×48px), a wide block for the value, and a narrow block for the label. The card skeleton should respect border-radius: var(--radius-xl) and padding: var(--space-lg) to match the real card's footprint exactly. Zero CLS on load.

6 · Accessibility 6 issues

This is the most critical category. The entire icon system is built on emoji — 20+ unique emoji across the sidebar, stats cards, topbar, and notifications. Emoji are announced verbatim by screen readers ("chart increasing emoji, Dashboard"), making the navigation experience deeply confusing. Three of the six issues below are directly caused by this single architectural choice.
Category Screen / Component Issue Severity v0-Standard Fix
Emoji Icons Sidebar Navigation
dashboard/layout.js · all item.icon		 All 30+ sidebar navigation icons are emoji characters (📊, 👤, 🔔, 🎯, 📝, 🎉, 📄, 🏢, 🎓 etc.). Screen readers announce these as their full CLDR description: a VoiceOver user hears "chart increasing emoji, Dashboard" for every nav item. Additionally, emoji render at different sizes across OSes (Windows renders 📊 smaller than macOS), and they do not respond to the parent element's color, breaking dark mode icon tinting. High Replace all emoji with an SVG icon library — Lucide React (npm i lucide-react) is the v0/Vercel standard. Each icon becomes: <LayoutDashboard size={18} aria-hidden="true" />. The aria-hidden="true" removes icons from the accessibility tree; the adjacent text label provides the name. SVGs inherit currentColor, so active/hover/dark mode color changes work automatically. Lucide covers all required icons.
Emoji Icons Topbar Buttons
dashboard/layout.js · theme toggle, bell		 The theme toggle button contains 🌙 or ☀️ as its sole content. Screen readers announce: "crescent moon button". The notification bell is 🔔 — announced as "bell button". Neither button has an aria-label. The mobile hamburger uses ☰ (trigram for heaven Unicode) with no label. All three fail the WCAG 1.1.1 non-text content criterion. High Add aria-label to each: aria-label={theme === 'light' ? 'Switch to dark mode' : 'Switch to light mode'}. Replace emoji with SVG icons (<Moon size={18} />, <Bell size={18} />, <Menu size={18} /> from Lucide) wrapped in aria-hidden="true". The aria-label on the button provides the accessible name.
Active State Sidebar Navigation
dashboard/layout.js		 Active navigation items are identified via className={`sidebar-link ${pathname === item.href ? 'active' : ''}`}. The CSS class changes visual style but does not communicate state to assistive technologies. Screen reader users cannot identify which page they are currently on from the navigation alone — violating WCAG 4.1.2. High Add aria-current="page" to the active link: aria-current={pathname === item.href ? 'page' : undefined}. Screen readers announce "current page" for the active item. This is a one-line fix. Additionally, ensure the sidebar has role="navigation" and aria-label="Main navigation" on the <nav> element.
Skip Link Root Layout / Dashboard
layout.js		 There is no skip-to-main-content link as the first focusable element. Keyboard-only users must Tab through the entire sidebar (30+ links across 4 sections) to reach the main content area on every page load. This is a fundamental keyboard navigation failure and a WCAG 2.4.1 bypass blocks violation. Medium Add as the very first element in the dashboard layout: <a href="#main-content" className="skip-link">Skip to main content</a>. Add CSS in globals.css: .skip-link { position: absolute; transform: translateY(-100%); } .skip-link:focus { transform: translateY(0); } — visually hidden by default, appears on Tab. Add id="main-content" to the <main> element.
Form Labels Login Form
login/page.js · form inputs		 The email input uses id="login-email" but the label does not have a matching htmlFor="login-email" — the label uses the class .form-label but has no htmlFor attribute visible in the source. Without this association, clicking the label does not focus the input, and screen readers do not announce the label when the input receives focus. Medium Add htmlFor="login-email" to the email label and htmlFor="login-password" to the password label. Verify the same for all forms across the application. As a systemic fix, create a <FormField id label required hint /> wrapper component that automatically wires htmlFor ↔ id, preventing this class of error in future.
Live Region Login · All Pages
error/success feedback		 Error and success messages appear in the DOM after user actions (login failure, credential fill) but are rendered as regular <div> elements. Screen readers will not announce these messages because they are not in a live region — a user navigating by keyboard will have no indication that the login failed unless they manually navigate to the error element. Low Add role="alert" to error messages (announces assertively — appropriate for failures) and role="status" to success/info messages (announces politely — appropriate for confirmations). The <FormAlert> component recommended in Section 4 should set these roles automatically based on variant. Toast notifications should use role="status" for info/success and role="alert" for errors.

7 · Visual Polish & Design Consistency 4 issues

The card system, badge variants, gradient top-borders on stats cards, and glassmorphism effects are all polished and well-implemented. What holds the application back from a v0-quality finish is a cluster of precision issues: a double-layer avatar in the topbar, the absence of reduced-motion support (covered in Section 3), and the emoji icon system which makes the sidebar feel like a prototype rather than a production product.
Category Screen / Component Issue Severity v0-Standard Fix
Double Layer Topbar Identity
dashboard/layout.js · avatar wrapper		 The topbar entity identity wraps an EntityLogo inside a <div className="avatar avatar-sm">. The .avatar class renders initials as a fallback background; EntityLogo also renders initials as its own fallback. The result is two overlapping gradient/initial backgrounds until the image loads — creating a flash of double-initials. The EntityLogo's internal border-radius also conflicts with the outer .avatar border-radius. High Remove the .avatar wrapper entirely — EntityLogo already handles its own shape, size, fallback gradient, and initials. Use <EntityLogo name={…} size="sm" shape="rounded" /> directly. The component's size="sm" maps to 32px, matching the topbar's space. Delete the redundant width: '32px', height: '32px', overflow: 'hidden' inline styles.
Z-Index Stack Mobile Overlay
dashboard/layout.js		 The mobile sidebar overlay uses zIndex: 99 as a hardcoded inline value. The notification panel dropdown has no explicit z-index, relying on DOM order. The sidebar uses CSS class-based positioning. With three overlapping layers (sidebar, notification panel, overlay) using mixed positioning strategies, the stacking order is unpredictable — especially if a dropdown opens while the mobile overlay is visible. Medium Define the z-index stack as CSS custom properties: --z-sidebar: 200; --z-dropdown: 300; --z-overlay: 400; --z-modal: 500; --z-toast: 600. Apply these via CSS classes, never inline. The mobile overlay should use var(--z-overlay), the sidebar var(--z-sidebar), and the notification dropdown var(--z-dropdown). This makes the full stack legible in one place.
Inline Style Creep All Dashboard Pages
student/page.js and others		 The student dashboard alone contains 23 inline style={{}} objects for spacing, colors, and font sizes. Some duplicate values already defined as CSS classes (e.g., style={{ marginBottom: '1.5rem' }} when .mb-6 or a layout token could serve). Inline styles are harder to audit, cannot be overridden by dark mode selectors or media queries without !important, and create an inconsistent authoring pattern. Medium Establish a rule: inline styles are only acceptable for dynamic values (e.g., computed widths from data, progress bar percentage). Static spacing, colors, and typography should use CSS classes. Add utility classes to globals.css: .mb-4, .mb-6, .gap-3, .text-sm, .font-semibold. Migrate static inline styles to these classes. This reduces bundle size and enforces the token system.
Icon Consistency Stats Cards
student/page.js · stats-card-icon		 Stats card icons use emoji at font-size: 1.5rem inside a 48×48px container. The visual size of emoji at 1.5rem varies by operating system: on Windows, emoji render at approximately 20×20px optical size; on macOS/iOS they render larger and more saturated. This creates inconsistent perceived icon size between users — a stats card row will look balanced on macOS and misaligned on Windows. Low Replace emoji in stats cards with SVG icons sized explicitly at 24px × 24px. Use Lucide: FileText, CheckCircle, Gift, Calendar for the student dashboard stats. Set size={24} strokeWidth={1.75} and pass the semantic color token via the icon's color prop or CSS currentColor inheritance from the parent's color class.

8 · Security UX 3 issues

Security UX failures are the highest-risk category for user trust. Two issues here are directly on the login page — the entry point every user sees first. The type="text" password field and plaintext demo credentials together signal that this application treats credential security as unimportant, which undermines trust even in a SaaS demo context.
Category Screen / Component Issue Severity v0-Standard Fix
Credential Exposure Login Page
login/page.js		 type="text" on the password field means: (1) passwords are visible to shoulder-surfers, (2) the browser does not offer to save credentials to the password manager, (3) autocomplete tools fill the wrong field type, (4) any browser extension that logs form values will capture the password as a plain text input value. This is the most impactful security-UX failure in the application. High Change to type="password" with autoComplete="current-password". Add a show/hide toggle (covered in Section 4). This is a one-word fix with zero UX cost. The demo flow works identically — users click a card, credentials fill, they click Sign In. Whether the password is masked or not does not impede the demo login flow.
JSON.parse Risk Topbar · Employer Role
dashboard/layout.js · activeCampus		 JSON.parse(sessionStorage.getItem('activeCampus')) is called directly in the render function with no try/catch. If sessionStorage is unavailable (SSR context), contains a corrupted value, or returns null, this throws an uncaught exception that crashes the entire layout — making every employer-role page inaccessible. The code also calls typeof window !== 'undefined' but the JSON.parse still runs server-side in some edge cases. High Wrap in a safe accessor: const getActiveCampus = () => { try { const raw = sessionStorage?.getItem('activeCampus'); return raw ? JSON.parse(raw) : null; } catch { return null; } }. Call this in a useMemo with typeof window !== 'undefined' as a guard. Additionally, migrate active campus state to a React context or URL param rather than sessionStorage — it's more reliable and survives tab clones.
Error Message Login Page
login/page.js · handleSubmit error		 setError(result.error) directly surfaces the raw NextAuth error string. NextAuth credential errors can expose strings like "No user found with this email" or "CredentialsSignin" depending on the provider implementation. User enumeration attacks use login error specificity to determine whether an email is registered. The exact string depends on the auth.js implementation, which was not audited, but the surface exists. Medium Replace with a generic message regardless of error type: setError('Invalid email or password. Please try again.'). Never reflect result.error directly. If specific errors need to be handled (e.g., account locked), map them client-side from non-sensitive error codes: if (result.error === 'AccountLocked') setError('Account temporarily locked…') — no email-existence information should be revealed.

What Is Working Well

Area Strength Observed
Design Token System The globals.css CSS custom property architecture is comprehensive and well-named: semantic color layers (--bg-primary, --text-secondary), a complete spacing scale, border radius scale, shadow elevation set, and transition tokens. Dark mode is implemented via a [data-theme='dark'] attribute with full token overrides — the right approach.
Dark Mode Dark mode is user-controllable (not system-only), persisted to localStorage, and applied via a CSS attribute selector on <html>. The color system flips correctly across backgrounds, text, borders, and shadows. Stats card icon backgrounds have explicit dark variants. This is implemented better than most production applications.
EntityLogo Component The layered fallback chain (explicit URL → Clearbit API → deterministic gradient + initials) is elegant and prevents broken image states entirely. The imgLoaded state pattern (gradient always rendered underneath, image shown on onLoad) is the correct approach. Consistent initials + gradient across all users is a strong visual identity pattern.
Button System The .btn system in globals.css is solid: 5 variants, 4 sizes, disabled state, active press scale (scale(0.97)), and hover lift on primary/success/danger. The gradient backgrounds on primary, success, and danger buttons with inner highlight (inset 0 1px 0 rgba(255,255,255,0.1)) are a high-polish detail that lifts the application above most SaaS templates.
Empty States The three-tier empty state distinction (no-data empty, zero-results empty, error-state) is architecturally understood and partially implemented in the student dashboard. The .empty-state CSS class with icon, title, and description is defined and applied. This is a detail most teams omit entirely.
Role-Based Architecture The sidebar menu config driven by a menuConfig object keyed by role is clean and maintainable. Role-based routing (auto-redirect after auth to the correct dashboard) is correctly implemented. The 41-page application handles 4 distinct user roles with appropriate page sets — a substantial and well-organized application scope.
SWR Integration Using SWR for client-side data fetching with skeleton fallback is the correct modern pattern. The data-fetching structure (route → /api/[role]/dashboard → PostgreSQL) is clean. The formatLPA, formatDate, formatStatus, getStatusColor utility functions centralize display logic correctly.
