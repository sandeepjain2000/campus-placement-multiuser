Infrastructure Page Wireframe Analysis
=====================================

Scope
-----
This note analyzes the older wireframe / preview blocks still present on the College Infrastructure page.

Files reviewed:

  src/app/dashboard/college/infrastructure/page.js
  src/app/api/college/infrastructure/route.js
  src/app/globals.css


Summary
-------
The Infrastructure page is partly real and partly still a prototype.

Real data currently comes from:

  - college_facilities
  - college_calendar where event_type = 'placement_drive'
  - /api/college/settings for the college website URL

However, the calendar, website, and social sections still use wireframe/skeleton UI structures. Some visible text has been softened from "wireframe" or "dummy" to "preview", but the implementation is still based on wireframe classes and fake shell blocks.

Finding:
Infrastructure booking metadata is stored as JSON inside college_calendar.description.

Fields stored inside description include:

  - roomId
  - roomName
  - startTime
  - endTime
  - company
  - notes
  - channels

Problem:
This is brittle and makes infrastructure bookings hard to query, validate, migrate, and report on.

Expected behavior:
Create a dedicated infrastructure booking table or add structured columns for booking metadata. If JSON must be used, store it in a jsonb metadata column instead of overloading a description text field.


Recommended Remediation
-----------------------
1. Convert the calendar into a real Infrastructure Calendar:

   - Default to the current month.
   - Use real production class names.
   - Keep showing saved bookings.
   - Remove wireframe terminology and CSS from this section.

2. Replace Website Preview with Website Announcements:

   - Show configured website URL from College Settings.
   - Show bookings tagged for Web as real announcement cards.
   - Remove browser chrome and skeleton blocks.
   - Hide the section or show an empty state when no website URL exists.

3. Replace Social Preview with Channel Distribution:

   - Load saved social URLs from College Settings.
   - Show each configured channel as a real destination.
   - Show tagged bookings/events per channel.
   - Remove skeleton social-post shells.
   - Do not imply posting or feed integration unless those integrations exist.

4. Rename channel toggles:

   Replace "Show in previews" with a production label such as "Announcement destinations" or "Communication channels".

5. Clean CSS:

   Remove or isolate:

   - wireframe-panel
   - wireframe-block
   - wireframe-social-post
   - wireframe-browser-chrome
   - wireframe-calendar-cell
   - sw-mock-* styles

   Production screens should not depend on CSS comments or class names that describe dummy UI.

6. Improve the data model:

   Move infrastructure metadata out of college_calendar.description into structured columns or a dedicated booking table.


Bottom Line
-----------
The Infrastructure page is no longer purely dummy because bookings and channel selections do persist. However, the calendar, website, and social sections are still presented through old wireframe/skeleton UI. They should be converted into real data-backed production sections or hidden until the required integrations exist.
